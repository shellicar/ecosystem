## tsconfig.check.json is a symlink

Every package has `tsconfig.check.json` as a symlink to `../../tsconfig.check.json`. The `Find` tool does not follow symlinks, so the file appears missing when scanning a package directory. Use `ls -la` to detect it. The symlink was created with `ln -s ../../tsconfig.check.json packages/core-di-lite/tsconfig.check.json`.

## types.ts / interfaces.ts circular import is intentional

`types.ts` imports `IServiceProvider` (for `InstanceFactory<T>`). `interfaces.ts` imports types from `types.ts`. TypeScript handles circular `import type` correctly. `verbatimModuleSyntax: true` requires `import type` on both sides. This matches the same pattern in core-di.

## `injectDependencies` reads from `instance.constructor`, not the descriptor

When a factory is used (`to(IFoo, factory)`), the instance is constructed by the factory, not by `new Implementation()`. Reading `@dependsOn` metadata from the descriptor's implementation class would give the wrong class. Reading from `(instance as object).constructor` gets the actual concrete class, which is where the decorator writes metadata. This handles factory registrations correctly.

## `resolve` during build routes through `resolveInternal`

`ServiceProvider.resolve()` calls `resolveInternal()`. After `build()` completes, all registered services are in the singletons map, so `resolveInternal` becomes a pure map lookup. The indirection means factories can call `x.resolve()` during `buildProvider()` and trigger eager resolution of their dependencies in the correct order.

## Circular dep detection covers both factory deps and `@dependsOn` deps

The `resolving` Set tracks in-flight identifiers across both factory calls and property injection. Self-dependency via `@dependsOn(ISelf)` works: when `injectDependencies` calls `resolveInternal(ISelf)` for the instance currently being built, `ISelf` is still in the resolving set, so `CircularDependencyError` is thrown. Build-time detection means no deferred failures.

## `ServiceCreationError` uses `{ name: string }` not `ServiceRegistration<T>`

core-di's `ServiceCreationError` takes a `ServiceRegistration<T>` which this package doesn't have. The lite version uses `{ name: string }` directly. Enough for error messages, no extra type needed.
