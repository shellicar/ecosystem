# core-di @abraham/reflection removal

## inject.ts is not just a side effect

The prompt targets `metadata.ts`. Three files actually need touching:

1. `src/private/metadata.ts` - the WeakMap replacement
2. `inject.ts` at the package root - this is the tsup inject entry, prepended to every bundle. It imported `@abraham/reflection` to bootstrap the polyfill before any bundle code ran. After the dependency is removed, the import must be cleared. The file should remain (tsconfig still references it); leaving it empty is correct.
3. `package.json` / `pnpm-lock.yaml` via `pnpm remove`

Missing the `inject.ts` change leaves a dead import and a broken build with no obvious connection to `metadata.ts`.

## Why the WeakMap replacement is safe

`@abraham/reflection` `getMetadata` walked the prototype chain. The WeakMap replacement does not. This is safe because both callers (`dependsOn.ts` and `ServiceProvider.ts`) always pass the class constructor directly, never an instance or subclass. The prototype walk was never exercised.

## Older CHANGELOG entries have bare version tags

The changelog generator uses whatever `tag` field is in each release entry. Releases before the `core-di@X.Y.Z` tag format was adopted used bare `3.1.6`-style values. Those links will not resolve to real tags. This is pre-existing; do not touch it.

## core-di-lite design

### Why the builder chain's rationale disappears without lifetime

In core-di, the builder chain (`register().to().singleton()`) exists because `.singleton()/.transient()/.scoped()` are the meaningful terminal steps. Without lifetime, `.to()` is the only call after `register()`, and it becomes a terminal that returns `void`. That asymmetry is visible to callers. The design options differ primarily on whether to keep that chain (Option 1), collapse it to a single call (Option 2), or remove class-direct registration entirely (Option 3).

### Eager build mechanics are the same across all three options

All options share the same underlying build: `buildProvider()` resolves every registered identifier, filling a singleton map. The resolution algorithm is identical to core-di's with lifetimes stripped. Circular dep detection still needs a ResolutionContext equivalent, just without the lifetime maps. After build, `provider.resolve()` is a map lookup.

### The `@dependsOn` sharing question has a concrete edge case

With two separate WeakMap stores (one in core-di, one in core-di-lite), a class decorated with `@dependsOn` whose metadata was written via one package's store is invisible to the other. This only matters if a class is used with both containers. Whether that scenario is realistic determines whether extraction to a shared package is worth adding a third published package.

## core-di-lite Phase 1 (Red)

### Package structure

Created `packages/core-di-lite/` from scratch. `tsconfig.check.json` in each package is a symlink to `../../tsconfig.check.json` (the root file). The `Find` tool does not follow symlinks so it appeared the file was missing from other packages. Use `ls -la` to detect symlinks. Created the symlink with `ln -s ../../tsconfig.check.json packages/core-di-lite/tsconfig.check.json` to match the pattern.

### Type design

`types.ts` and `interfaces.ts` have a circular `import type` dependency (same as core-di). `types.ts` imports `IServiceProvider` for `InstanceFactory<T>`. `interfaces.ts` imports `ServiceBuilderOptions`, `ServiceIdentifier`, `SourceType` from types. TypeScript handles circular `import type` correctly. `verbatimModuleSyntax: true` in the base tsconfig requires `import type` for type-only imports, which both sides use.

`ServiceBuilderOptions<T>` returns `void` (not `ILifetimeBuilder`) since `.to()` is terminal in core-di-lite.

`ServiceCreationError` uses `{ name: string }` for the `implementation` parameter instead of `ServiceRegistration<T>`, avoiding the need for a `ServiceRegistration` type alias in a package that doesn't have the concept.

### Tests

14 tests in `test/core-di-lite.spec.ts`. All fail with `Error: not implemented` since `createServiceCollection()` throws. Build and type-check both pass.

Test counts: 14 failed / 0 passed.

### One subtle test detail

Tests 12 and 13 (circular dependency) check that `services.buildProvider()` throws, not `provider.resolve()`. This is correct for an eager-build container: all errors surface at build time. The test stub causes them to fail at `createServiceCollection()` before even reaching `buildProvider()`. The implementation in Phase 2 needs to throw during `buildProvider()` itself.
