The `'isNull'` branch and the generic fallback were left as-is â€” `isNull` intentionally pushes no parameter, the fallback already handled the remaining operators correctly. ## The version race â€” why this released as 1.1.5, not 1.1.4

Phase 3 bumped to `1.1.4` against `origin/main` at the time. Between Phase 3 and Phase 4, PR #34 (security-audit release batch) merged to `main` and shipped `@shellicar/cosmos-query-builder@1.1.4` with the changelog "Updated patch dependencies". Same package, same version, different content. By the time the Courier ran the pre-push smoke test, the version number was gone.

The Courier brief's `CONFLICTING` flow (fetch + merge `origin/main`) is shaped for an unrelated package's batch landing alongside; it doesn't resolve this. The smoke test stopped the cast before push. The SC merged `origin/main`, the working tree was reconciled (whereOr `fixed` entry moved past the merged-in `1.1.4` release marker so it became Unreleased, `CHANGELOG.md` regenerated to show `1.1.4` as just "Updated patch dependencies" matching what's actually published), and the brief's Phase 4 was re-dispatched with an explicit Redo block bumping to `1.1.5`.

**For Phase 5**: `1.1.4` is published and untouchable. The release that ships the whereOr fix is `cosmos-query-builder@1.1.5`. The tag and changelog content are already correct in the merged base; `gh release create cosmos-query-builder@1.1.5` is what triggers npm-publish.

## Trap: the changelog generator is cwd-relative

`scripts/src/generate-changelog.ts` resolves the package path against `process.cwd()`, not the repo root. `pnpm --filter scripts changelog packages/<pkg>` puts cwd inside `scripts/`, so a relative argument resolves to `scripts/packages/<pkg>` (nonexistent) and the script fails. Two ways through:

- Pass an absolute path (used in Phase 3).
- Run from `scripts/` with `pnpm exec tsx src/generate-changelog.ts <absolute-path>` (used in the Phase 4 Redo, matching the brief's revised form).

Validator (`pnpm exec tsx src/validate-changes.ts` from `scripts/`) is also fine.

This is worth knowing because if a future Phase 3 or reconciliation step uses the original relative form from the skill example, it fails with a path-not-found that doesn't say "this is the cwd resolution". The fix is the invocation, not the script.

## Pattern worth surfacing to the PM/FM

Same-package same-version collision is its own class of conflict, distinct from "unrelated package's release batch landed alongside". The Phase 3 bump number computed at brief-write becomes stale if `origin/main` moves before Courier ships. Two mitigations the SC's Phase 4 verification already named:

- Phase 3 brief specifies the bump as a delta ("next patch over `origin/main` at the moment of execution") rather than a fixed number.
- Phase 4 preflight adds a same-package release-plumbing check ("is the package's current `package.json` version already tagged on `origin/main`?"), turning this into a stop-and-flag rather than a smoke-test discovery.

This cast caught it at smoke-test; the cost was one wasted ship attempt, not a bad release.
