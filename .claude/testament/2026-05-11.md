## build-clean entrypoints

`src/index.ts` only exports the default plugin. Type information (`Feature`, `Options`, `ILogger`, `Features`) is accessed via the `./types` sub-path: `import type { Options } from '@shellicar/build-clean/types'`. The main entrypoint is the unplugin plugin object, not bundler-specific — bundler-specific sub-paths (`./vite`, `./esbuild`, etc.) re-export the plugin's bundler adapter.

## build-version virtual module

The `./version` sub-path is the consumer-side import (`import versionInfo from '@shellicar/build-version/version'`). The default export is a `VersionInfo` object populated at build time. The plugin (`@shellicar/build-version`) injects the real values. Without the plugin, the module returns an object with all empty strings.

## core-di vs core-di-lite

The key distinction is not just CLI vs full-stack. It's eager vs configurable:
- `core-di-lite` builds all services at `buildProvider()` time, singletons only, no lifetimes, no scopes, no `resolveAll`, throws `DuplicateRegistrationError` if you register the same identifier twice.
- `core-di` has lifetimes (`Lifetime` enum), scoped providers, `resolveAll`, `IServiceModule`, `Symbol.dispose`, and `overrideLifetime`.

## cosmos-query-builder Container type

The package defines its own local `Container` interface (a reduced version of `@azure/cosmos`) to avoid SDK version lock-in. You don't pass the full SDK `Container` directly — you pass something satisfying the local interface (which the real SDK container satisfies). The `@azure/cosmos` SDK is a peer dep, not bundled.

## winston-azure-application-insights optional deps

The two AI SDK packages are aliased as `applicationinsightsv2` and `applicationinsightsv3` (using pnpm's `npm:` alias). Importing the real AI client requires these installed. The `ITelemetryClientV2` and `ITelemetryClientV3` interfaces in `src/public/` can be used to create mocks or adapters without the actual SDK.
