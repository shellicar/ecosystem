### Issues found

1. `Object.entries()` in `injectDependencies` silently drops symbol-keyed `@dependsOn` fields. `MetadataType` allows `string | symbol` keys, `dependsOn` stores `ctx.name` which can be a symbol, but `Object.entries` only iterates strings. core-di has the same behavior, so this is inherited, not introduced. Low severity since symbol-named class fields in practice are rare.

2. No test for registration-order independence. The chained `@dependsOn` test registers services in dependency order (C, B, A). If someone registers A first and C last, the eager build should still resolve correctly because `resolveInternal` recurses. The implementation handles this, but there's no test proving it.

3. `MetadataType` and `AbstractNewable` are exported from `src/index.ts` but aren't part of any public API signature. They're implementation details in the public surface.

## Phase 1 (Red) for factory singleton identity bug (#27)

### What was done

Created `packages/core-di/test/factory-singleton.spec.ts` with 5 tests across 4 describe blocks.

### How the singleton map works

The singleton cache is a `Map<ServiceRegistration<T>, T>` (i.e., keyed by the `implementation` field on `ServiceDescriptor`). When `register(IFoo).to(MyClass, factory).singleton()` is called, `descriptor.implementation` is set to `MyClass` regardless of which factory was provided. So two registrations with the same class but different factories share the same singleton key.

`ResolutionContext.getFromLifetime` checks this map on resolve. If the key already exists, it returns the cached instance without calling `createInstance`. That is why the second factory never runs.

### Test results

- Type-check: passes
- Tests 1-2 (existing behaviour preserved): pass
  - "same implementation class resolves to the same instance" (no factory)
  - "same factory reference resolves to the same instance"
- Tests 3-5 (bug, expected to fail): fail
  - "different factories resolve to different instances"
  - "each interface consistently returns its own instance"
  - "factory and non-factory resolve to different instances"
- All 66 existing tests across 22 other test files: pass

### Where the fix will go

The fix is in how the singleton map is keyed. The key needs to incorporate factory identity so that different factory references produce different keys. The relevant code path is:
- `ServiceProvider.resolveInternal` calls `context.getFromLifetime(descriptor.implementation, ...)`
- `ResolutionContext.getFromLifetime/setForLifetime` use `descriptor.implementation` as the Map key

The `ServiceDescriptor` already has `createInstance` which wraps the factory, but `implementation` is what drives the cache key. The descriptor itself could be used as the key (each `createDescriptor` call produces a fresh object), or a composite key incorporating the factory function reference.

## Phase 2 (Green) for factory singleton identity bug (#27)

### The fix

Added a `cacheKey` field to `ServiceDescriptor`. The key is set at registration time: `factory ?? implementation`. When no factory is provided, the cache key is the implementation class (preserving existing behavior where same-class registrations share a singleton). When a factory is provided, the cache key is the factory function reference itself.

This works because:
- Same class, no factory: `cacheKey = MyClass` for both registrations, same key, shared singleton
- Same class, same factory ref: `cacheKey = sameRef`, shared singleton
- Same class, different factory refs: `cacheKey = fn1` vs `cacheKey = fn2`, separate singletons
- One with factory, one without: `cacheKey = fn` vs `cacheKey = MyClass`, separate singletons

The `cacheKey` survives `{ ...descriptor }` shallow clone in `ServiceCollection.clone()`, so scoped providers still correctly find singletons from the parent scope.

### Files changed

- `types.ts`: Added `cacheKey: object` to `ServiceDescriptor`, changed `RegistrationMap` key type from `ServiceRegistration<T>` to `object`
- `ServiceBuilder.ts`: Set `cacheKey: factory ?? implementation` in `createDescriptor`
- `ServiceProvider.ts`: Use `descriptor.cacheKey` instead of `descriptor.implementation` for cache get/set. Explicit `<T>` type parameter needed on `getFromLifetime` call because `object` key doesn't let TypeScript infer the return type
- `ResolutionContext.ts`: Changed `getFromLifetime`/`setForLifetime` parameter from `ServiceRegistration<T>` to `object` to match the new key type

Circular dependency detection (`markResolving`/`unmarkResolving`) still uses `descriptor.implementation` because that is about the class, not the factory.

### Why not use the descriptor itself as key

`ServiceCollection.clone()` does `{ ...descriptor }` which creates a new object. If the descriptor were the key, cloned descriptors in scoped providers wouldn't match the singleton map entries from the parent scope, breaking singleton behavior across scopes.
