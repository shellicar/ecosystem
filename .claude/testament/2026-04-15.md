## Code review of core-di-lite (Phase 4)

Reviewed all source and test files in `packages/core-di-lite/` against core-di reference and the design report.

### What was done well

The worker correctly simplified core-di's patterns rather than copying them verbatim. The string-keyed metadata store (`WeakMap<object, Map<string, unknown>>`) became a direct `WeakMap<object, MetadataType>`. The `constants.ts` file was dropped because there's no metadata key to name. `ResolutionContext` was inlined as a `resolving` Set on `ServiceProvider`. Lifetime-related code is entirely absent. The `DuplicateRegistrationError` was a good addition not in the original prompt.

### Issues found

1. `Object.entries()` in `injectDependencies` silently drops symbol-keyed `@dependsOn` fields. `MetadataType` allows `string | symbol` keys, `dependsOn` stores `ctx.name` which can be a symbol, but `Object.entries` only iterates strings. core-di has the same behavior, so this is inherited, not introduced. Low severity since symbol-named class fields in practice are rare.

2. No test for registration-order independence. The chained `@dependsOn` test registers services in dependency order (C, B, A). If someone registers A first and C last, the eager build should still resolve correctly because `resolveInternal` recurses. The implementation handles this, but there's no test proving it.

3. `MetadataType` and `AbstractNewable` are exported from `src/index.ts` but aren't part of any public API signature. They're implementation details in the public surface.

## Factory singleton identity bug (#27)

### Root cause

The singleton cache in `ResolutionContext` is a `Map` keyed by `descriptor.implementation` (the class). When `register(IFoo).to(MyClass, factoryA).singleton()` and `register(IBar).to(MyClass, factoryB).singleton()` both register, both descriptors have `implementation = MyClass`. The second resolve hits the cache and returns the first factory's instance without ever calling `factoryB`.

### Why the fix uses `cacheKey = factory ?? implementation`

The obvious alternative is using the descriptor object itself as the cache key. That breaks because `ServiceCollection.clone()` does `{ ...descriptor }` (shallow clone), producing a new object reference. Scoped providers inherit their parent's singleton map, so cloned descriptors would miss cache entries from the parent scope.

Using `factory ?? implementation` means:
- No factory: key is the class (existing shared-singleton behavior preserved)
- With factory: key is the function reference (different factory refs get separate singletons, same ref shares)

### Trap: circular dependency detection

`markResolving`/`unmarkResolving` must stay keyed by `descriptor.implementation`, not `cacheKey`. Circular dependency is about the class being constructed, not which factory constructs it.
