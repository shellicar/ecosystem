## Factory singleton identity bug (#27)

### Root cause

The singleton cache in `ResolutionContext` is a `Map` keyed by `descriptor.implementation` (the class). When `register(IFoo).to(MyClass, factoryA).singleton()` and `register(IBar).to(MyClass, factoryB).singleton()` both register, both descriptors have `implementation = MyClass`. The second resolve hits the cache and returns the first factory's instance without ever calling `factoryB`.

### Why the fix uses `cacheKey = factory ?? implementation`

The obvious alternative is using the descriptor object itself as the cache key. That breaks because `ServiceCollection.clone()` does `{ ...descriptor }` (shallow clone), producing a new object reference. Scoped providers inherit their parent's singleton map, so cloned descriptors would miss cache entries from the parent scope.

Using `factory ?? implementation` means:
- No factory: key is the class (existing shared-singleton behavior preserved)
- With factory: key is the function reference (different factory refs get separate singletons, same ref shares)

### Trap: circular dependency detection

`markResolving`/`unmarkResolving` must stay keyed by `descriptor.implementation`, not `cacheKey`. Circular dependency is about the class being constructed, not which factory constructs it.

## Syncpack configuration and dependency hygiene

### Vite 8 OXC decorator bug

Vite 8 uses OXC for transforms. OXC cannot parse the `@dependsOn` decorator combined with definite assignment (`!`) syntax used in core-di. `SyntaxError: Invalid or unexpected token` on all test files that use `@dependsOn(X) private readonly field!: T`. Vite 7 works fine. This blocks vite 8 adoption until OXC fixes it.

Dangerous interaction: vitest resolves vite as a peer dependency. If vite 8 ends up anywhere in the lockfile (even from a temporary `pnpm add`), it can leak into vitest's peer resolution for packages that don't declare vite directly. `pnpm why vite` is essential to verify only one version is resolved. `pnpm add vite@^7` to revert a specifier does NOT remove vite 8 from peer resolutions in the lockfile.

### Syncpack ignore rules

Version groups are evaluated in order (first match wins). Ignores for specific packages/deps must go before the default group.

- Pinned fixture versions (`@azure/cosmos` in cosmos451/cosmos460, `applicationinsights` in v2/v3_03/v3_09/v3_12): `isIgnored: true` scoped by `packages` and `dependencies`
- Peer deps ignored globally: they use intentionally wider ranges than devDeps
- `unplugin`: cross-major split (v2 in build-clean/build-version, v3 in build-graphql), ignored until aligned
- Non-standard dep types (overrides, resolutions, optionalDependencies): ignored

### Specifier format consistency

`scripts/package.json` and `packages/typescript-config/package.json` were the source of wide `^N` specifiers (e.g. `^4`, `^25`, `^22`) that caused syncpack mismatches against the `^N.x.y` format used everywhere else. `pnpm add pkg@^N` writes the short form; `pnpm add pkg@^N.x.y` writes the exact form.

### Examples referencing npm versions instead of workspace

Many examples had `"@shellicar/build-clean": "^1.2.3"` (npm version) instead of `"workspace:^"`. This caused pnpm to resolve the published package's peer metadata (e.g. `vite: "^6"`) instead of the local workspace version. Always use `workspace:^` for local package references in examples.

### Remaining syncpack issues

- `workspace:*` vs `workspace:^` in 5 files (build-version examples, core-config, graphql-codegen-treeshake, winston-azure-application-insights)
- `@nuxt/schema` pinned at `3.21.2` (no `^`) in build-graphql: intentional
- `typescript` pinned at `5.9.3` (no `^`) in build-graphql: intentional

## Phase 5: core-di-lite review fixes

### Symbol-keyed @dependsOn silently fails

`dependsOn` stores metadata with `ctx.name` as the key, which is `string | symbol`. The `MetadataType` is `Record<string | symbol, ...>`, so the metadata store handles symbols correctly. But `injectDependencies` uses `Object.entries(deps)` to iterate, which only yields string-keyed properties. Symbol keys are silently dropped.

The test creates a class with `@dependsOn(IDep) readonly [symField]!: IDep` where `symField` is a `Symbol`. It asserts that the property remains `undefined` after injection (the `run()` method returns `'not injected'` via optional chaining fallback). This documents the current behaviour without fixing it.

### Registration-order independence

The existing chain test (A depends on B depends on C) registers bottom-up (C, B, A). The new test registers top-down (A, B, C). `resolveInternal` handles this naturally because when it encounters A, it calls `injectDependencies` which calls `resolveInternal(IB)`, which recursively resolves IC first. The `singletons` map and `resolving` set make this work regardless of iteration order.

### Removed public exports

`MetadataType` and `AbstractNewable` removed from `src/index.ts`. `MetadataType` is only used internally in `metadata.ts` (imports directly from `../types`). `AbstractNewable` is defined but not referenced anywhere in the package.

### Symbol injection fix was already in place

The SC had already updated `injectDependencies` to use `Reflect.ownKeys` instead of `Object.entries`. The old code was left as a comment. The test was initially written to assert the broken behaviour (`'not injected'`), then updated to assert the working behaviour (`'dep'`). Cleaned up the commented-out `Object.entries` code from `ServiceProvider.ts`.

## Phase 7: Red - factory singleton bug

### The prompt's prediction vs reality

The prompt said: tests 1-2 (same class/factory) must pass, tests 3-5 (different factories) must fail. This assumed core-di-lite had the same implementation-class-keyed cache as core-di (buggy).

The actual implementation (post Phase 6 refactor) caches by `identifier`, not by `implementation`. `ServiceProviderBuilder.singletons` is `Map<ServiceIdentifier, instance>`. Each registered identifier gets its own entry regardless of implementation class.

Result: the bug is different. Tests 1-2 FAIL (different identifiers with same class get separate instances when they should share). Tests 3-5 PASS (different factories correctly yield different instances).

### The actual bug

Two registrations pointing at the same implementation class each create their own instance. `register(IServiceA).to(SharedImpl)` and `register(IServiceB).to(SharedImpl)` produce two separate `SharedImpl` instances. The intent (from core-di's design) is that they should share one.

Similarly, `register(IServiceA).to(SharedImpl, factory)` and `register(IServiceB).to(SharedImpl, factory)` where `factory` is the same function reference should share one instance. Currently they don't.

## Phase 8: Green - factory singleton fix

### What the fix does

Three files changed:

1. `ServiceDescriptor.ts`: added `factory?: InstanceFactory<T>` field. Previously the factory was captured only in the `createInstance` closure and not reachable from outside.

2. `ServiceBuilder.ts`: stores `factory` on the descriptor alongside the existing `createInstance` closure. One line added.

3. `ServiceProviderBuilder.ts`: added `sharedSingletons: Map<object, any>` alongside the existing `singletons: Map<ServiceIdentifier, any>`. Before creating a new instance, computes `cacheKey = descriptor.factory ?? descriptor.implementation` and checks `sharedSingletons`. If found, stores the shared instance in `singletons` (by identifier, so `ServiceProvider.resolve()` can find it) and returns. If not found, creates normally and stores in both maps.

### The two maps

`singletons` is keyed by identifier: it's the map `ServiceProvider` receives and uses for `resolve()` lookups. Every identifier must have an entry.

`sharedSingletons` is keyed by `factory ?? implementation`: it's the sharing mechanism. Same key = same instance. Separate from `singletons` because they have different key spaces.

### Circular dependency detection is unaffected

`resolving` stays keyed by `identifier`. The `sharedSingletons` check is before `resolving.add(identifier)`, which is correct: if we already have the shared instance, there's no construction happening, so no circularity to detect.
