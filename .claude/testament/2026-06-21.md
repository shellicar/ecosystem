## Trap: trustPolicy: no-downgrade trips install, and the exclude is keyed by package NAME

pnpm 11's `trustPolicy: no-downgrade` failed install (`ERR_PNPM_TRUST_DOWNGRADE`) on five existing lockfile entries: chokidar@4.0.3, rollup@3.30.0, semver@5.7.2, semver@6.3.1, ua-parser-js@1.0.41. These are pre-existing deps whose trust evidence dropped, not anything this mission introduced. The mission anticipated this: add a top-level `trustPolicyExclude` (sibling of `trustPolicy`, NOT nested under `auditConfig`) naming only the flagged packages.

The non-obvious part: `trustPolicyExclude` is keyed by package name, so two separate `semver@...` lines collapse to one and only one version gets excluded. After adding all five as separate lines, install still tripped on `semver@6.3.1` alone. The fix is the documented multi-version form â€” a single entry combining versions with `||`: `'semver@5.7.2 || 6.3.1'` (pnpm docs show the same for webpack). After that, install is clean.

If a future install trips on a NEW package here, the same pattern applies: name it in `trustPolicyExclude`, and if it's a second version of a package already listed, fold it into that entry with `||` rather than adding a second line.
