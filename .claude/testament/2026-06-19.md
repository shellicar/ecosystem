## What shipped: optional peer on the real `applicationinsights` name

`@shellicar/winston-azure-application-insights` is bring-your-own-client: the consumer
passes in the Application Insights client they already installed. Both AI majors used to
sit in `optionalDependencies`, which installs by default, so consumers got *both* v2 and
v3 pulled into their graph even though they use one. Released as **6.0.8** (patch).

The shipped shape:

```json
"peerDependencies": { "applicationinsights": "^2 || ^3", ... },
"peerDependenciesMeta": { "applicationinsights": { "optional": true } }
```

The two aliases (`applicationinsightsv2`, `applicationinsightsv3`) moved to
`devDependencies` so the package still builds, type-checks, and tests against both majors.

## Trap 1: aliased `npm:` specifiers are illegal in `peerDependencies`

The mission's original design peered the *aliases* (`"applicationinsightsv2":
"npm:applicationinsights@^2"`). pnpm rejects that with
`ERR_PNPM_INVALID_PEER_DEPENDENCY_SPECIFICATION` — peer specifiers must be a plain semver
range, `workspace:`, or `catalog:`, never an `npm:` alias. The fix the SC chose live: peer
the *real* package name spanning both majors, drop the alias from the peer entirely. The
consumer installs `applicationinsights`, so the real name is what satisfies the peer.

## Trap 2: a runtime alias import had to become type-only

The source imported a *value*, `import { KnownSeverityLevel } from 'applicationinsightsv3'`,
which compiles into the shipped `.js` as a literal `import ... from "applicationinsightsv3"`
— a name no consumer can resolve once the alias leaves the package's own deps.
`KnownSeverityLevel` is a string enum, so it became `import type` plus the severity map cast
to the enum. The shipped `.js` now has zero alias references.

## The `.d.ts` alias leak is intentional and harmless

The shipped `.d.ts` still imports types from the aliases (`ITelemetryClientV2/V3`). That does
not break consumers: the ecosystem builds with `skipLibCheck`, so the unresolved alias names
in our declarations are never checked. Don't try to scrub them.
